# Care Signal: Strategic Reality and Implementation Roadmap

**Status:** Canonical strategy and execution roadmap
**Date:** 2026-04-23
**Audience:** Leadership, engineering, product, clinical governance
**Context:** This document outlines the strategic intent, current operational reality, and the non-negotiable roadmap to transform Care Signal from a hollow UI shell into a functional, data-driven pillar of the Paeds Resus platform.

---

## 1. Strategic Intent: What Care Signal Must Be

According to the Platform Source of Truth (PSOT) and the Fellowship Qualification guidelines, Care Signal is a **first-class product** designed for healthcare providers. Its core jobs are:

1. **Provider-Facing QI Reporting:** A confidential space for clinical staff to log incidents, near-misses, and system gaps (knowledge, resources, leadership, etc.) encountered during pediatric emergencies.
2. **Fellowship Discipline (Pillar C):** The automated engine that tracks a provider's monthly reporting discipline (24 consecutive qualifying months, with strict grace and catch-up rules).
3. **Internal Operational Intelligence:** The source of truth for facility-level gaps, feeding into institutional dashboards and future accreditation models.

**Crucially, Care Signal is NOT Safe-Truth.** Safe-Truth is for parents and guardians; Care Signal is for providers. The two data streams must never mix in user-facing metrics.

---

## 2. Current Reality: The "Hollow Shell" Problem

An exhaustive audit of the codebase reveals a critical disconnect between the PSOT's intent and the current implementation. While the frontend presents a sophisticated analytics experience, the backend is almost entirely mocked.

### 2.1 What Works
- **Submission Persistence:** The `logEvent` mutation correctly writes submissions to the `careSignalEvents` table and fires a `care_signal_submission_created` analytics event.
- **Fellowship Streak Engine:** The underlying logic for calculating the Care Signal streak (including grace periods and resets) is mathematically sound in `fellowship-care-signal-streak.ts`.

### 2.2 What is Broken or Missing
- **Mocked Analytics:** The `CareSignalAnalytics.tsx` page is entirely theatre. It relies on hard-coded constants for total incidents, gap percentages, and outcome distributions. There are no live tRPC connections.
- **Stubbed Backend:** The `care-signal-events.ts` router returns empty arrays for `getEventHistory`, zeros for `getEventStats`, and static strings for `getRecommendations`.
- **Borrowed Insights:** The widgets on the Care Signal page (`ResourceGapWidget` and `MultiFacilityBenchmarkWidget`) do not actually query Care Signal data. They query `resus_resource_gap` events generated by ResusGPS.
- **Fellowship UI Misinformation:** The `FellowshipProgress.tsx` page incorrectly states that users must submit "≥3 events per month" as a blanket rule, contradicting the PSOT's baseline of ≥1 event (with ≥3 only required for catch-up).
- **Timezone Bug:** The streak calculation groups events using UTC (`getUTCFullYear()`, `getUTCMonth()`), directly violating the PSOT's mandate to use East Africa Time (EAT).
- **Thin Data Model:** The `careSignalEvents` table lacks critical fields like `facilityId`, reviewer tracking, and structured eligibility markers, making institutional review and facility-level analytics impossible.

## 3. Implementation Roadmap: Closing the Gap

To make Care Signal deliver on its intended value, we must execute the following roadmap. Priority is given to establishing the feedback loop: providers must see their own data reflected back to them.

### Phase 1: Establish the Feedback Loop (Immediate Priority)
*Providers will not continue to report if their data disappears into a void.*

1. **Wire `getEventHistory`:** Update the router to query the `careSignalEvents` table and return the authenticated user's actual submission history.
2. **Wire `getEventStats`:** Calculate basic stats (total events, most common gap, etc.) from the user's actual data in the database.
3. **Fix the Fellowship UI Copy:** Correct the text in `FellowshipProgress.tsx` to accurately reflect the ≥1 normal month threshold and the ≥3 catch-up threshold.
4. **Fix the Timezone Bug:** Refactor `calculateCareSignalPillar` in `fellowship.ts` to bucket events strictly by EAT (UTC+3) calendar months.

### Phase 2: Data Model and Analytics Reality
*Replace the theatre with actual data-driven insights.*

1. **Schema Migration:** Add `facilityId` to the `careSignalEvents` table to enable institutional filtering. Add fields for review status and reviewer notes.
2. **Wire `getGapAnalysis`:** Implement real aggregation of the `systemGaps` JSON array from the user's submissions to power the gap analysis charts.
3. **Connect `CareSignalAnalytics.tsx`:** Remove all hard-coded mock data. Wire the page to use the newly implemented tRPC queries for stats, history, and gap analysis.
4. **Dynamic Recommendations:** Replace the static strings in `getRecommendations` with a rules engine that maps specific reported gaps to actionable, context-aware advice.

### Phase 3: Institutional Intelligence and Review
*Deliver value to hospitals and platform administrators.*

1. **Implement `submitForReview`:** Update the mutation to actually change the event status in the database and notify the relevant institutional admin.
2. **Facility Dashboards:** Create views for institutional admins to see aggregated, anonymized Care Signal data for their specific `facilityId`.
3. **Disentangle ResusGPS Data:** Clarify the UI so providers understand the difference between ResusGPS resource gap telemetry and deliberate Care Signal incident reporting.

---

## 4. Conclusion

Care Signal currently presents the appearance of a mature QI tool without the underlying data infrastructure to support it. By executing this roadmap—starting with simply returning the user's own history—we will transform Care Signal from a hollow form into the powerful, behavior-changing engine the Paeds Resus platform requires.
